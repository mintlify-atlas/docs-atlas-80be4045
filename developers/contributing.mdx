We welcome contributions from the community! Whether it's bug fixes, new features, or documentation improvements, your help makes iStory better.

## Getting Started

<Steps>
  <Step title="Fork the repository">
    Fork the [iStory repository](https://github.com/your-org/web3_Ai_eStory) on GitHub
  </Step>
  
  <Step title="Clone your fork">
    ```bash
    git clone https://github.com/<your-username>/web3_Ai_eStory.git
    cd i_story_dapp
    ```
  </Step>
  
  <Step title="Install dependencies">
    ```bash
    npm install
    ```
  </Step>
  
  <Step title="Create a feature branch">
    ```bash
    git checkout -b feature/your-short-description
    ```
  </Step>
</Steps>

### Commit Messages

Follow [Conventional Commits](https://www.conventionalcommits.org/) style:

<CodeGroup>
```bash Feature
feat(record): add real-time transcription indicator
```

```bash Fix
fix(api): handle empty audio uploads gracefully
```

```bash Documentation
docs: update contributing guidelines
```

```bash Breaking Change
feat(auth)!: migrate to new authentication flow

BREAKING CHANGE: Old auth tokens are no longer valid
```
</CodeGroup>

<Note>
The format is: `type(scope): description`. Common types: `feat`, `fix`, `docs`, `style`, `refactor`, `test`, `chore`.
</Note>

## Code Style & Standards

### Linting

The project uses ESLint and TypeScript. Run these before submitting:

```bash
npm run lint        # Check for errors
npm run lint:fix    # Auto-fix issues
```

### TypeScript Guidelines

<AccordionGroup>
  <Accordion title="Always define types" icon="t">
    ```typescript
    // ❌ Bad
    function saveStory(data: any) { ... }
    
    // ✅ Good
    interface StoryData {
      title: string;
      content: string;
      mood?: string;
    }
    
    function saveStory(data: StoryData) { ... }
    ```
  </Accordion>
  
  <Accordion title="Use interfaces for objects" icon="shapes">
    ```typescript
    // Define in app/types/index.ts
    export interface Story {
      id: string;
      title: string;
      content: string;
      author_id: string;
      created_at: string;
    }
    ```
  </Accordion>
  
  <Accordion title="Avoid type assertions" icon="shield">
    ```typescript
    // ❌ Avoid
    const data = response as StoryData;
    
    // ✅ Better - validate at runtime
    import { z } from 'zod';
    const StorySchema = z.object({
      title: z.string(),
      content: z.string(),
    });
    const data = StorySchema.parse(response);
    ```
  </Accordion>
</AccordionGroup>

### Component Organization

```typescript
// 1. Imports
import { useState } from "react";
import { Button } from "@/components/ui/button";

// 2. Types/Interfaces
interface Props {
  title: string;
  onSave: () => void;
}

// 3. Component
export function StoryCard({ title, onSave }: Props) {
  // 4. Hooks
  const [isLiked, setIsLiked] = useState(false);
  
  // 5. Event handlers
  const handleLike = () => {
    setIsLiked(!isLiked);
  };
  
  // 6. Render
  return (
    <div className="story-card">
      <h3>{title}</h3>
      <Button onClick={handleLike}>
        {isLiked ? "Liked" : "Like"}
      </Button>
    </div>
  );
}
```

## Testing Requirements

### Writing Tests

All new features should include tests:

<CodeGroup>
```typescript Component Test
import { render, screen } from "@testing-library/react";
import { describe, it, expect } from "vitest";
import { StoryCard } from "../components/StoryCard";

describe("StoryCard", () => {
  it("renders story title", () => {
    render(<StoryCard title="My Story" onSave={() => {}} />);
    expect(screen.getByText("My Story")).toBeInTheDocument();
  });
});
```

```typescript API Route Test
import { describe, it, expect, vi } from "vitest";
import { POST } from "../app/api/journal/save/route";

vi.mock("@/lib/auth", () => ({
  validateAuthOrReject: vi.fn().mockResolvedValue("user-123"),
}));

describe("POST /api/journal/save", () => {
  it("saves story successfully", async () => {
    const req = new Request("http://localhost:3000/api/journal/save", {
      method: "POST",
      body: JSON.stringify({ title: "Test", content: "Content" }),
    });
    
    const response = await POST(req);
    const data = await response.json();
    
    expect(response.status).toBe(200);
    expect(data.success).toBe(true);
  });
});
```
</CodeGroup>

### Running Tests

Before submitting your PR:

```bash
npx vitest run          # Unit tests
npx playwright test     # E2E tests
npm run build          # Production build
```

<Warning>
All tests must pass before your PR will be reviewed. Add new tests for new features.
</Warning>

## Pull Request Process

<Steps>
  <Step title="Ensure quality checks pass">
    - [ ] Linting passes (`npm run lint`)
    - [ ] All tests pass (`npx vitest run`)
    - [ ] Production build succeeds (`npm run build`)
    - [ ] Code follows existing patterns
  </Step>
  
  <Step title="Update documentation">
    If you've changed:
    - API routes → Update `docs/API_REFERENCE.md`
    - Database schema → Update `docs/DATABASE_SCHEMA.md`
    - Testing patterns → Update `docs/TESTING_GUIDE.md`
  </Step>
  
  <Step title="Create pull request">
    Push your branch and open a PR with:
    - Clear title describing the change
    - Detailed description of what and why
    - Screenshots for UI changes
    - Link to related issues
  </Step>
  
  <Step title="Address review feedback">
    Respond to reviewer comments and make requested changes
  </Step>
</Steps>

### PR Template

```markdown
## Description
Brief summary of changes and motivation.

## Type of Change
- [ ] Bug fix (non-breaking change which fixes an issue)
- [ ] New feature (non-breaking change which adds functionality)
- [ ] Breaking change (fix or feature that would cause existing functionality to not work as expected)
- [ ] Documentation update

## How Has This Been Tested?
- [ ] Unit tests
- [ ] E2E tests
- [ ] Manual testing

## Screenshots (if applicable)

## Checklist
- [ ] Code follows the project's style guidelines
- [ ] Self-review completed
- [ ] Commented hard-to-understand areas
- [ ] Documentation updated
- [ ] No new warnings generated
- [ ] Tests added and passing
- [ ] Dependent changes merged and published
```

### Before Starting Major Work

<Note>
For significant features or breaking changes, **open an issue first** to discuss the design and scope. This prevents wasted effort if the approach isn't aligned with the project direction.
</Note>

### Code of Conduct

Be respectful and inclusive. Follow the guidelines in `CODE_OF_CONDUCT.md`:

- Use welcoming and inclusive language
- Be respectful of differing viewpoints
- Accept constructive criticism gracefully
- Focus on what's best for the community
- Show empathy towards other community members

## Security Issues

<Warning>
**Do NOT open public issues for security vulnerabilities.** Instead, email the maintainers at `security@estory.app` with:
- Detailed description of the vulnerability
- Steps to reproduce
- Potential impact
- Suggested fix (if any)
</Warning>

Allow time for the team to respond and patch before public disclosure.
